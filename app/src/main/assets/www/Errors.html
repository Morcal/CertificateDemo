1. Errors

PWS的错误系统设计和实现

1.1. Error Scopes

错误系统的设定是根据系统涉及的业务范围和内容决定的。
主要包括

  • Portal协议交互(对象为NAS/BRAS)
  • Portal 服务（主要是认证授权）
  • REST服务

错误系统的划分和组成是与业务模块的划分和组成相对应的。

1.2. Exceptions Flow

系统业务模块对应的错误系统大致流向为：

1.3. Exception Handling

其中任何业务（包括以上提到的三个方向）均遵照一般的java惯例进行设计和实现，即
API和调用均返回正常结果表达，非正常结果（包括portal服务体系中的错误）均以异常抛出。

  • REST API authentication/authorization相关的错误由spring-security以及在spring-security基础上创建的servlet filter, controller interceptor抛出异常。最后在RestAuthenticationEntryPoint中将相关的异常包装为JSON返回至客户端。
  • 服务层产生的非正常结果，抛出业务相关的异常
  • 底层支持（包括与NAS/BRAS进行portal协议通信，RADIUS服务器/客户端)产生非正常结果时抛出底层异常，在服务层将底层异常封装为服务层异常，再抛出。
  • 异常向上抛出至controller（controller为与客户端交互最外层）时不处理，统一在controller advisor中将以上抛出的异常包装为JSON返回值客户端。

所以的unchecked异常如果抛出至controller则在，controller advisor中处理。
内部服务unchecked异常需要在每处可能发生异常的地方自行处理。

1.4. Exception Content

由于系统定义了全局的错误代码，所以系统中定义的异常（主要是checked异常）均需要携带全局
定义的错误代码。实现上所有系统定义的异常都从抽象的顶层异常基础，该顶层异常中包含了
PortalError成员，该成员类中定义了error code，和error text。
错误处理中需要从异常中获取错误代码，并将其包装在返回至客户端的错误数据（JSON）中。
